# 基于证据的实现与调试

## 基本循环

```text
观察事实
→ 提出一个可证伪假设
→ 做最小修改
→ 执行能区分该假设的测试
→ 保存结果
→ 接受、否定或升级阻塞
```

## 证据优先级

1. 当前目标 commit 的代码；
2. 当前模型 config、参考实现和 checkpoint；
3. 当前环境的命令、日志和测试输出；
4. 对应版本的官方文档或测试；
5. 已确认知识库；
6. 历史经验。

越低优先级的材料越不能覆盖当前直接证据。

## 有效假设

有效：

```text
当前权重 loader 只识别 gate_up_proj，而 checkpoint 使用 w1/w3；
若增加明确映射，missing keys 应从两组减少到零。
```

无效：

```text
可能是权重问题，尝试改一下 loader。
```

## 防止重复尝试

执行前比较：

- 假设是否与历史相同；
- 修改文件是否相同；
- 命令和参数是否相同；
- 输入 revision 是否变化；
- 是否新增了能改变判断的证据。

只有最后两项至少一项成立时，才允许重复相同命令。

## 日志阅读

至少保留：

- 完整命令和工作目录；
- stdout/stderr；
- 退出码和 timeout；
- 首个确定失败点；
- 最后一个成功步骤；
- 稳定错误签名；
- 与前一次的差异。

不要只截取最后一行错误。

## 最小修改

最小修改指：

- 只覆盖当前 capability；
- 可通过单独测试验证；
- 不引入隐式 fallback；
- 不复制大段上游代码来回避局部差异；
- 不混入性能重构。
